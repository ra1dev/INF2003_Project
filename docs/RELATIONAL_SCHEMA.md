# EPL Analytics Relational Schema

This package implements the normalized relational database for the INF2003 EPL analytics project.

## Files

- `sql/01_schema.sql` creates the PostgreSQL schema, constraints, and indexes.
- `scripts/build_relational_data.py` reads the raw CSV files and writes cleaned load files to `data/processed/`.
- `sql/02_load_processed.sql` loads the generated CSV files into PostgreSQL.
- `sql/03_validation.sql` runs integrity checks after loading.
- `data/reports/build_report.json` is generated by the ETL and documents row counts and source limitations.

## Design Notes

The schema separates player season statistics from player-club membership:

- `player_season_stats` stores season-level player performance facts.
- `player_team_season` stores club membership by player and season.

This is necessary because some players appear for multiple clubs in the same season. Therefore, the dependency `player_id, season_id -> team_id` is false and placing `team_id` inside `player_season_stats` would create an update anomaly.

As a data-cleaning policy, final `player_season_stats` only includes player-season rows that have at least one reliable club membership in `player_team_season`. Unmapped player stat rows are preserved in staging and reported in `data/reports/dropped_unmapped_player_stat_rows.json`.

The final schema keeps percentage stats that are present in the source dataset:

- `tackle_success_pct`
- `cross_accuracy_pct`
- `shooting_accuracy_pct`

Raw rows are preserved in staging tables before transformation:

- `staging_match_raw`
- `staging_player_stats_raw`
- `staging_player_club_raw`

## Scope Decision

The final relational scope is `2015/16` through `2021/22`. The original player-stat files include `2022/23` and `2023/24`, but the teammate enrichment file `teammate_dump/pl_player.csv` does not provide reliable player-club membership after `2021/22`. Those later seasons are therefore excluded from the final normalized relational tables to keep the schema internally consistent and defensible.

The raw source data also contains a few football-rule anomalies:

- one match-team row has shots on target greater than total shots,
- four player-season rows have appearances greater than 38.

The raw values are preserved in staging. The final clean tables handle them as follows:

- the Newcastle vs West Ham match on `2021-08-15` corrects West Ham shots from `8` to `18`, documented in `data/manual_corrections.json`;
- the four impossible player-season rows are rejected from `player_season_stats` and documented in `data/reports/rejected_player_stat_rows.json`.

This allows the final schema to keep hard constraints such as `shots_on_target <= shots` and `appearances BETWEEN 0 AND 38`.
